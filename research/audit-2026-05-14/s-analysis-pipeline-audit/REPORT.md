# Analysis Pipeline Audit — 2026-05-14

**Scope:** Read-only investigation of the analysis pipeline, raw data capture, condition handling, comp engine learning, and schema readiness for V2 features.
**Auditor:** Claude Code (claude-sonnet-4-6)
**No code was modified during this audit.**

---

## 1. Executive Summary

- **"No modifications reported" is real analysis output**, not a hardcoded placeholder. It is produced by the `modifications` findings rule, which always fires — either as a caution ("Modifications noted") or a positive ("No modifications reported"). Every listing gets one of these two findings regardless of content.
- **Raw description text is explicitly and deliberately discarded** at parse time. The architecture is "facts-only": extract signals → store booleans → drop the text. The `mentioned_*` boolean columns are the only persisted trace of description content. The `raw_html_snapshot_key` column exists in the schema but is never populated.
- **The schema is partially ready for V2** but description-dependent features (condition flag extraction, confidence-scored disclosure analysis) cannot be retrofitted without either re-fetching listings or a decision to preserve raw text going forward.
- **The `/analyze/[id]/full` page does not exist** — it is a redirect stub back to `/analyze/[id]`. The TeaserBlock's "Read the full analysis" CTA leads nowhere new.
- **`comp_engine_runs` is well-structured for backtesting** but `actual_price` is never populated in production runs, and per-factor scores are not serialized to the DB, limiting attribution analysis.

## 2. Section A: Full Analysis Card Behavior

### Which component renders it?

`components/analyze/TeaserBlock.tsx`. It is mounted in `app/(app)/analyze/[id]/page.tsx` at line 210 and receives `analysisRow` (a `listing_analyses` DB row or null), `listingId`, and `viewerTier`.

### Where does the findings count come from?

```
analysisRow?.finding_count ?? 0
```
`finding_count` is an integer column on the `listing_analyses` table, written by `POST /api/analyze` as `findings.length` at analysis time. It is **a real count stored in the database**, not computed at render time. If `analysisResult.data` is null (no analysis row for this listing), `findingCount` defaults to 0 and the headline reads "Continue with the full analysis." rather than "We found X things."

### Where does "No modifications reported" come from?

It is a **real finding generated by the `modifications` rule** in `lib/findings/rules/modifications.ts`. This rule:
1. Scans `listing.description` for a keyword list (modified, aftermarket, engine swap, coilover, etc.) using word-boundary regex
2. Also checks `listing.modification_notes` (always null currently — never populated)
3. **Always returns a finding.** If no keywords match: returns `{ severity: 'positive', title: 'No modifications reported', body: 'No modifications or aftermarket parts are mentioned in the listing...' }`. If keywords match: returns a caution finding instead.

This is genuine analysis — not a default empty-state string — but it is derived purely from the absence of keywords in the stored description text. A listing description with zero modification language will always produce this positive finding.

> **Key implication:** "No modifications reported" means "no modification keywords found in the seller's description at the time the URL was submitted." It does not mean "we confirmed the car is unmodified."

### What gates the sign-in blur (paywall behavior)?

Two mechanisms:
1. **CTA link**: For `viewerTier === 'anonymous'`, the "Read the full analysis" button routes to `/signup?next=/analyze/{id}/full` instead of the full-analysis page.
2. **CSS blur on finding body text**: The `finding.body` paragraph has `blur-[2.5px] select-none` applied. This is client-side CSS only — the text content IS sent to the browser in the HTML and is accessible in page source, just visually obscured.

`viewerTier` is derived server-side by `getViewerTier()` and passed as a prop.

Note: **The `/analyze/[id]/full` page is currently a redirect stub** (`app/(app)/analyze/[id]/full/page.tsx`) that immediately redirects back to `/analyze/[id]`. Authenticated users who click "Read the full analysis" are sent back to the same page.

### Full data flow trace: URL paste → "Full Analysis" card

```
1. User pastes URL → POST /api/analyze
   ├── parseBatHtml(html, url) runs in memory
   │   └── CanonicalListing assembled including .description (JSON-LD prose, in-memory only)
   ├── decodeVin() runs if VIN found
   ├── matchGeneration() determines generation_id
   ├── extractSourceMentions(listing.description) → mentioned_* booleans (description DISCARDED after this)
   ├── listings upsert: structured fields written, description NOT included in row
   ├── runFindingsRules({ listing, generationId }) → Finding[] (uses listing.description still in memory)
   │   Rules: mileageVsAverage (dormant), serviceRecords, modifications, titleStatus, generationWatchout
   ├── listing_analyses INSERT: { findings: Finding[], finding_count: N, analysis_data: {...} }
   └── computeCompsV2(listingId) → comp_engine_runs INSERT

2. GET /analyze/[id] (Server Component)
   ├── listings SELECT * WHERE id = params.id
   ├── generation_editorial SELECT WHERE generation_id = listing.generation_id AND content_status = 'verified'
   ├── listing_analyses SELECT WHERE listing_id ORDER BY created_at DESC LIMIT 1
   ├── computeCompsV2(listing.id) — recomputes fresh every page view (not cached)
   ├── getRecallsByMakeModelYear() → NHTSA API
   ├── matchDefects(context) → catalog items from static markdown files
   └── TeaserBlock rendered with analysisRow (which has findings + finding_count)

3. TeaserBlock renders
   ├── findings = parseFindings(analysisRow?.findings).slice(0, 3)
   ├── findingCount = analysisRow?.finding_count ?? 0
   └── Headline + blurred finding cards rendered
```

### Is raw listing description preserved anywhere?

**No.** The `CanonicalListing.description` field holds the full JSON-LD description prose in memory during the analysis pipeline, but the analyze route's `row` object explicitly omits it:

> `// Raw description is not stored — source-mention signals are extracted above and the text is discarded here (facts-only architecture).` — `app/api/analyze/route.ts` line 173

The `raw_html_snapshot_key` column exists in `listings` and is intended as a pointer to an R2 HTML snapshot, but **nothing in the current codebase writes to it**. The `app/api/analyze/route.ts` row construction never includes it.

### Source-mention signals (structured seller claim capture)

These are the closest thing to preserved description content. Set as booleans at parse time, then the text is dropped:

| Signal | Column | Populated? | Meaning of `true` |
|---|---|---|---|
| Repaint | `mentioned_repaint` | Yes | Pattern matched in description |
| Accident history | `mentioned_accident_history` | Yes | Any accident phrase found |
| Engine service | `mentioned_engine_service` | Yes | Engine work phrase found |
| Transmission service | `mentioned_transmission_service` | Yes | Clutch/gearbox work phrase found |
| Matching numbers | `mentioned_matching_numbers` | Yes | "numbers matching" or similar |
| Modifications | `mentioned_modifications` | Yes | Aftermarket/upgrade/custom etc. |
| Recent PPI | `mentioned_recent_ppi` | Yes | PPI or pre-purchase inspection |
| Original owner | `mentioned_original_owner` | Yes | One-owner/from new/since new |

**Critical nuance:** Only `true` is stored; `null` means "not mentioned." There is no `false` (= "explicitly denied"). "Accident-free" maps to `mentioned_accident_history = true`, same as "was in an accident." The column name means "topic appeared in text" not "claim direction."

## 4. Section C: Current Condition / Claim Handling

### Does any code extract condition signals from seller text?

Yes, at two levels:

**Level 1 — Disclosure extraction (BaT parser, `lib/listing-parser/bring-a-trailer.ts`):**
- `extractPaintDisclosure()` scans description for repaint/respray/color-change phrases → structured result (is_repainted, original_exterior_color, repaint_year, repaint_disclosure sentence)
- `extractReupholsteryDisclosure()` scans for reupholstery phrases → structured result
- These are genuine description-text analyses, but they run at parse time and the results are stored as discrete columns — not the text itself

**Level 2 — Boolean signal extraction (`lib/extractors/source-mentions.ts`):**
- `extractSourceMentions()` runs 8 pattern-match passes over description text
- Returns boolean signals for 8 disclosure categories
- These are stored in the `mentioned_*` columns

**Level 3 — Findings rules (`lib/findings/rules/`):**
- `modifications.ts`: keyword scan of `listing.description` (checked at analysis time, before description is discarded)
- `serviceRecords.ts`: keyword scan of `listing.description`
- `generationWatchout.ts`: keyword scan of `listing.description` (IMS-specific, but uses hardcoded generation IDs `'996'` and `'997-gen1'` that don't match actual DB generation IDs like `'996.1'`, `'997.1'` — **this rule never fires in practice**)
- `titleStatus.ts`: phrase scan of `listing.description` and `listing.title`
- `mileageVsAverage.ts`: **dormant** — always returns null (missing reference data)

### The defect catalog — structured matching, not description analysis

`matchDefects()` in `lib/originality.ts` operates entirely on structured fields from the listings row:
- `generation_id`, `engine_family`, `year`, `mileage`, `trim`, `trim_category`, `body_style`

It does **not** read listing descriptions. The defect catalog is a static set of generation-specific known issues encoded as YAML blocks in markdown files. Matching is purely categorical.

### If a seller writes "recent respray in original Guards Red" — what reads it?

**At fetch time (within the analyze pipeline, before storage):**
- `extractPaintDisclosure(description)` in `parseBatHtml` — YES, this reads it. It would match the `/[A-Z][a-zA-Z]+(?:\s+[A-Z][a-zA-Z]+){0,2}\s+respray/` pattern (or the `refinished in / was repainted in` patterns). It would extract `is_repainted = true`, `current_color_from_disclosure = 'Guards Red'`.
- `extractSourceMentions(description, ...)` — YES. The `mentioned_repaint` pattern `/\brespray(ed)?\b/i` would match and set `mentioned_repaint = true`.
- `modifications` rule and `serviceRecords` rule also scan description, but would not match on this string.

**After the listing is stored:**
- The raw text is gone. Only `is_repainted = true`, `mentioned_repaint = true`, and the `repaint_disclosure` sentence are available.
- No code reads listing description text on the analyze page — only the pre-extracted signals.

**Answer: Yes, the string would be read once at parse time. After storage, only structured fields persist.**

### Transmission detection

Transmission is determined **primarily from the structured "Listing Details" `<ul>`** in BaT's HTML, not from title or prose description.

The key function is `isTransmissionBullet(text)` which requires at least one of: transaxle, transmission, gearbox, PDK, Tiptronic, "[N]-speed" pattern, or G50/G96/G81 gearbox code. This explicitly rejects standalone "Automatic" or "Manual" words to prevent false positives like "Automatic Climate Control" or "Manual sunroof."

The `normalizeTransmission()` function then normalizes to compact form: "6-Speed Manual (G50)", "7-Speed PDK", "Tiptronic S", etc.

There is **no fallback to description prose for transmission** — unlike exterior/interior color which has a 3-tier cascade including description and visible text. If the Listing Details `<ul>` doesn't have a transmission bullet, `transmission` is null.

## 5. Section D: Comp Engine Learning Behavior

### Does the comp engine track comps that "moved" in price over time?

**No.** `comp_engine_runs` is a prediction log, not a price-tracking log. It captures each run's inputs and outputs, but there is no mechanism to observe a comp listing's price changing over time. Comp listings are fixed once sold; there is no concept of price revision in the data model.

### What does `comp_engine_runs` persist per run?

```
subject_listing_id    UUID → listings FK
model_version         'v1.0' (constant)
generation_used       e.g. '997.2'
weights_json          Record<FactorName, number> — the generation weight config used
comps_used_json       Array of { listing_id, similarity_score, recency_weight, final_weight, price_cents }
predicted_median      cents
predicted_p25         cents  
predicted_p75         cents
confidence_score      0–100
methodology_text      human-readable string
actual_price          null in all production runs (only set by backtest harness)
was_backtest          false in all production runs
verdict               priced_fairly | priced_above | priced_below | insufficient_comps | etc.
cascade_level         1–6 | null
```

**What is NOT stored per run:**
- `subject_data`: Column exists in schema (`subject_data: Json | null`) but `logCompRun()` never writes it — always null
- Per-factor breakdown for each comp: `ScoredComp.factor_scores` (a `Partial<Record<FactorName, number>>`) is computed in memory but only `similarity_score`, `recency_weight`, and `final_weight` are serialized to `comps_used_json`
- The comp pool that was rejected (after hard filtering, before similarity scoring)

### Is there enough to backtest predictions?

**Structurally yes, practically no.** The schema has `actual_price` and `was_backtest` columns. The `logCompRun` function accepts `options.actualPrice` for backtest use. The backtest pathway would be:

1. Iterate sold listings with `final_price` set
2. Call `computeCompsV2` with that listing, passing `actual_price = listing.final_price_cents` and `was_backtest = true`
3. Compare `predicted_median` vs `actual_price` across runs

However, in production use today, `actual_price` is never populated — every production row has `actual_price = null`. A separate backtest harness script would be needed to run this.

### Is "this comp sold for more than predicted by X%" captured?

**No.** No mechanism exists to capture prediction vs. outcome delta. `actual_price` is the field designed for it, but it is never populated in production runs.

## 7. Surprises / Things to Know

### 1. The `generationWatchout` findings rule never fires

`lib/findings/rules/generation-watchout.ts` uses `IMS_AT_RISK_GENERATION_IDS = new Set(['996', '997', '997-gen1'])`. The actual `generation_id` values in the database are `'996.1'`, `'996.2'`, `'997.1'`, `'997.2'`. These never match. A 996 or 997 listing will never receive the IMS bearing finding from this rule. The defect catalog (`matchDefects`) does cover this issue correctly, but the findings rules layer misses it.

### 2. The `mileageVsAverage` findings rule is dormant

`lib/findings/rules/mileage-vs-average.ts` is one line: `return null`. Always. The TODO says it's waiting for `average_mileage_at_year` data in `porsche_generations`. The finding count in `listing_analyses` never includes a mileage finding.

### 3. `source-mentions` signals only record `true`, never `false`

The `extractSourceMentions` function maps keyword match → `true`, no match → `null`. This means "accident-free" and "has had an accident" both result in `mentioned_accident_history = true`. The boolean doesn't tell you which claim the seller made — only that the topic was mentioned. A negative claim ("never been in an accident") is indistinguishable from a positive one ("minor fender bender") in the current signal structure.

### 4. `listing_analyses` is sparsely populated — `comp_count` and `confidence_score` are always null

The `listing_analyses` table has `comp_count` and `confidence_score` columns, but the analyze route never writes them. Comp results live in `comp_engine_runs`, not in `listing_analyses`. These two tables are not joined or synced.

### 5. The full analysis page doesn't exist yet

`app/(app)/analyze/[id]/full/page.tsx` is a one-line redirect stub. The TeaserBlock CTA correctly routes authenticated users there, but they immediately land back on `/analyze/[id]`. This is documented as a placeholder but is easy to miss.

### 6. `subject_data` in `comp_engine_runs` is always null

The schema includes `subject_data: Json | null` on `comp_engine_runs`, presumably intended to snapshot the subject listing's spec at run time (for backtesting without re-fetching). The `logCompRun` function never writes it. Without this, a backtest must re-join against the `listings` table, which can't detect spec changes (e.g., if trim_category was later corrected).

### 7. Description text exists in memory during the full pipeline — rules use it

The findings rules and paint/upholstery extractors all run while `listing.description` is still in the `CanonicalListing` object in memory. The decision to not persist description does not prevent analysis during the request; it prevents analysis on subsequent views. Any condition-flag extraction that could happen at request time (once, on first submission) can in principle run today. It just can't be re-run later.

### 8. `raw_html_snapshot_key` is a schema promise, not a feature

The column was designed for an R2 snapshot approach. Decision D3 explicitly retained it as a pointer. But there is no R2 write anywhere in the analyze flow. The column is always null in practice. The data-handling docs treat the R2 snapshots as a future capability.

## 8. Open Questions

1. **Intent of `accident_history_stated`**: This enum column (none_stated/minor_stated/major_stated) is used by the comp engine's `conditionSimilarity` calculation, but is never populated by the analyze route (Decision D6 broke it). Are comp similarity scores currently using defaults (0.5 = unknown) for all listings? Confirm.

2. **IMS bearing finding**: The findings rules layer won't fire for 996/997 IMS risk because of the generation ID mismatch. Does the defect catalog adequately cover this? Should the `generationWatchout` rule be updated or removed?

3. **`mentioned_*` boolean direction**: Is the current "topic mentioned = true" schema adequate for V2 disclosure analysis, or should it be extended to capture claim direction (affirmed/denied)? This affects backward compatibility — changing the semantics of existing `true` values.

4. **R2 snapshot decision**: Is the `raw_html_snapshot_key` approach still the intended path for raw preservation, or has the "facts-only" architecture superseded it? Decision D3 says snapshots are out of scope — does that remain the policy?

5. **Backtest harness**: The schema supports backtesting (actual_price, was_backtest columns) but no production code populates `actual_price`. Is a backtest harness script in scope, and should `subject_data` be populated in production runs to enable it?

6. **`listing_analyses` vs `comp_engine_runs`**: These two tables store overlapping result data. `listing_analyses.comp_count` and `listing_analyses.confidence_score` are never written. Is the intent to eventually sync them, or are they intentionally separate?

## Summary

### Top 3 Findings

1. **"No modifications reported" is real output but always-on.** The `modifications` rule in `lib/findings/rules/modifications.ts` has no null return path — it always produces either a caution or a positive finding. "No modifications reported" is the positive branch, not an empty state. Every listing gets a modifications finding. The IMS bearing rule (`generationWatchout`) never fires due to a generation ID mismatch with DB values.

2. **Raw description is gone — by design.** The description is analyzed in-memory during the request, then explicitly discarded. The `mentioned_*` boolean columns are the only persisted trace. `raw_html_snapshot_key` exists in the schema but is never populated. Condition-flag extraction from descriptions requires a policy change or re-fetch, not just a schema migration.

3. **The comp engine is well-structured but has two persistent gaps.** `actual_price` is never set in production (makes backtesting impossible without a separate harness run), and `subject_data` is never written (makes debugging comp predictions against a snapshot of the subject spec impossible). Per-factor scores are computed but not serialized to the DB.

### Schema Readiness for V2 Features

| Feature | Readiness | Key Gap |
|---|---|---|
| Condition flag extraction | Not ready | No raw text; description discarded |
| Price attribution engine | Partial | `actual_price` never set; factor scores not persisted |
| Confidence-scored disclosure | Not ready | Boolean signals only, no direction, no confidence score |
| Raw description preservation | Not ready | Active architectural policy against it; R2 approach designed but unimplemented |
| Cross-site active-listing aggregation | Partial | Schema ready; only BaT parser exists; no background polling |
